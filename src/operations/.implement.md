# Operations Layer Phase Plan

## Overview

**Layer:** `src/operations/`  
**Architecture Constraint:** Operations can import from `src/devices/` and `src/core/` but NOT from `src/experiments/`  
**Dependency Flow:** Experiments → Operations → Devices

---

## Target Directory Structure

```
src/operations/
├── __init__.py
├── base.py
├── step_logger.py
├── safety_interlocks.py
├── temperature_control.py
├── flow_control.py
├── sample_management.py
├── data_acquisition.py
└── config/
    ├── defaults.yaml
    |── paths.yaml
    └── gas_calibrations/
        └── .gitkeep
```

## Phase 1: Foundation

**Files:** `base.py`, `__init__.py`, `config/defaults.yaml`, `paths.yaml`, `config/gas_calibrations/.gitkeep`

**Description:** Establish the base infrastructure that all operation modules will depend on. Includes base classes (`BaseOperation`, `OperationResult`), custom exceptions (`OperationError`, `OperationTimeoutError`, `SafetyViolationError`), and default configuration values.

### Success Criteria
- [ ] All files created without syntax errors
- [ ] `from src.operations import BaseOperation` works
- [ ] `defaults.yaml` and `paths.yaml` loads correctly with PyYAML

## Phase 2: Step Logger

**File:** `step_logger.py`

**Description:** Implement the central logging system that all operations will use. Outputs both `.json` (machine-readable) and `.md` (human-readable) formats. Includes `StepLogEntry` dataclass and `StepLogger` class with `log_step()` and `finalize()` methods.

### Success Criteria
- [ ] `StepLogger` can be instantiated with valid parameters
- [ ] `log_step()` creates `StepLogEntry` objects correctly
- [ ] `finalize()` writes both `step_log.json` and `step_log.md`
- [ ] JSON output is valid and parseable
- [ ] Markdown table renders correctly

---

## Phase 3: Safety Interlocks

**File:** `safety_interlocks.py`

**Description:** Implement safety checking system that operations must call before state-changing actions. Includes `SafetyLimits` dataclass and `SafetyInterlocks` class with individual check methods and `emergency_shutdown()` capability.

### Success Criteria
- [ ] `SafetyLimits` can be loaded from `defaults.yaml`
- [ ] All individual check methods return correct `(bool, str | None)` tuples
- [ ] `check_all_targets()` aggregates violations correctly
- [ ] `emergency_shutdown()` attempts to stop all devices

---

## Phase 4: Temperature Control

**File:** `temperature_control.py`

**Description:** Implement temperature control operations with CSV logging and markdown step summary. Supports multi-step temperature programs with configurable ramp rates, hold times, and thermal stabilization. Integrates with safety interlocks. The logic will need a class as we will want to pass parameters, read values, etc.

### Parameters

| Parameter | Type | Exposed/Hidden | Default |
|-----------|------|----------------|---------|
| `target_temps` | `list[float]` | Exposed | — |
| `ramp_rates` | `list[float]` | Exposed | — |
| `hold_times` | `list[float]` | Exposed | — |
| `thermal_stabilization_time` | `float` | Exposed | 60.0s |
| `poll_interval` | `float` | Hidden | 5.0s |
| `tolerance` | `float` | Hidden | ±2.0°C |
| `timeout` | `float` | Hidden | 3600.0s |
| `overshoot_limit` | `float` | Hidden | 10.0°C |

### Output Files
- `temperature_log.csv`: Columns `[datetime, target_temp, read_temp]` logged every 5 seconds
- `readme.md`: Table with columns `[Step ID, Target Temp (°C), Ramp Rate (°C/min), Hold Duration (min)]`

### Success Criteria
- [ ] Executes multi-step temperature programs
- [ ] Writes `temperature_log.csv` with 5-second interval readings
- [ ] Writes `readme.md` with step summary table
- [ ] Respects safety interlocks
- [ ] Handles abort requests gracefully
- [ ] Logs steps via `StepLogger` when provided

## Phase 6: Sample Management

**File:** `sample_management.py`

**Description:** Implement sample ID generation and mandatory material parameter validation. Auto-generates sample IDs with datetime stamps, creates experiment directory structure, and writes metadata files. The logic will need a class.

### MaterialParameters Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `user_label` | `str` | **Mandatory** | User-provided sample identifier |
| `material_type` | `str` | **Mandatory** | e.g., "catalyst", "support", "standard" |
| `mass_mg` | `float` | **Mandatory** | Sample mass in milligrams |
| `operator` | `str` | Optional | Who is running the experiment |
| `composition` | `str` | Optional | e.g., "5% Pt/Al2O3" |
| `mesh_size` | `str` | Optional | e.g., "60-80 mesh" |
| `pretreatment_history` | `str` | Optional | Prior treatments |
| `batch_id` | `str` | Optional | Synthesis batch reference |
| `notes` | `str` | Optional | Free-form notes |

### Auto-generated ID Format
```
{YYYYMMDD}_{HHMMSS}_{material_type}_{user_label}
Example: 20250110_143052_catalyst_PtAl2O3_batch7
```

### Output Files
- `sample_metadata.json`: Machine-readable sample information
- `readme.md`: Human-readable experiment summary

### Success Criteria
- [ ] `MaterialParameters` validates mandatory fields on creation
- [ ] `create_sample()` generates sample_id and experiment_id with datetime
- [ ] Creates experiment directory structure
- [ ] Writes `sample_metadata.json` and `readme.md`
- [ ] Sanitizes user input for safe filenames
